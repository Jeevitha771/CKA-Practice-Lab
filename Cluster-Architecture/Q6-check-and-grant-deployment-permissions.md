# Q6. Check and Grant Deployment Permissions to ServiceAccount

## Task
Check if ServiceAccount `app-sa` in namespace `production` can create deployments. If not, grant the necessary permissions.

---
This question introduces a fantastic two-part scenario: auditing existing permissions and then fixing a gap. It also quietly introduces one of the most common "gotchas" in Kubernetes RBAC: **API Groups**.


## Part 1: The Theory (What's New)

To explain this , we need to talk about where things live in Kubernetes.

### API Groups

* **What it is:** Kubernetes organizes its resources into different "families" called API Groups. So far, everything we have used (`pods`, `nodes`, `secrets`) belongs to the "core" group (represented as `""` in YAML). However, `deployments` belong to a different family called the **`apps`** API group.
* **The Analogy:** Think of the office building having different wings. The main lobby is the "core" group (where Pods live). But Deployments live in the "Apps Wing." If you write a job description for Deployments but forget to tell the security guard to check the Apps Wing, the rules won't work.
* **The CLI Magic:** The good news? When you use `kubectl` imperative commands in the terminal, it is smart enough to automatically look up that `deployments` live in the `apps` wing. If you write YAML by hand, you have to specify it yourself!

---

## Part 2: Step-by-Step Solution

This question asks us to *check* first, and then grant the permission if it's missing.

### Step 1: The Audit (Check the Permissions)

We use our trusty `auth can-i` command to check if the robot currently has the permission.

```bash
kubectl auth can-i create deployments --as=system:serviceaccount:production:app-sa -n production

```

**Expected Output:** `no`
*(Since we only gave `app-sa` permission to read pods in Q1, it definitely cannot create deployments).*

### Step 2: Create the Role

Since the answer was `no`, we need to create a job description that allows creating deployments in the `production` namespace. Let's call it `deploy-creator`.

```bash
kubectl create role deploy-creator \
  --verb=create \
  --resource=deployments \
  -n production

```

*(Pro-tip : If they were to look at the YAML generated by this command, they would see `apiGroups: ["apps"]` automatically added by Kubernetes!)*

### Step 3: Create the RoleBinding

Now, we hand this new job description to the existing `app-sa` ServiceAccount. We will call the binding `app-sa-deploy-binding`.

```bash
kubectl create rolebinding app-sa-deploy-binding \
  --role=deploy-creator \
  --serviceaccount=production:app-sa \
  -n production

```

*(Remind them again: `namespace:name` is required when binding to a ServiceAccount!)*

---

## Part 3: Verify the Fix

Now that we have created the Role and bound it to the ServiceAccount, we run the exact same audit command from Step 1 to prove we fixed the issue.

```bash
kubectl auth can-i create deployments --as=system:serviceaccount:production:app-sa -n production

```

**Expected Output:** `yes`

Here is the exact YAML that the `kubectl create role` command generated behind the scenes in the previous step.

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: deploy-creator
rules:
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["create"]

```

### The Breakdown

* **`apiGroups: ["apps"]`**: This is the crucial difference. In earlier examples with Pods and Secrets, this field was left as `[""]` (an empty string), which tells Kubernetes to look in the default "core" API group. Because Deployments were added to Kubernetes later in its lifecycle, they were placed in a separate API group named `apps`.
* **Multiple Groups**: If you ever need to create a single Role that manages both Pods and Deployments, you cannot combine them into a single rule block. You must create two separate items under `rules:` — one for the `""` group and one for the `"apps"` group.

If we attempts to write this YAML by hand on an exam and mistakenly puts `[""]` for Deployments, the API server will accept the file without throwing an error, but the permissions will silently fail to work because a "core Deployment" does not exist.

---

## Notes

- `deployments` belong to API group `apps` — `kubectl create role` handles this automatically
- To also manage ReplicaSets (which Deployments create), add `--resource=replicasets` to the same role
- Use `kubectl auth can-i --list` to see all permissions for a subject:

```bash
kubectl auth can-i --list \
  --as=system:serviceaccount:production:app-sa \
  -n production
```
