30 New Questions to ADD — Covering Missing 25% Domain

⚠️ This entire domain is absent from your Q57–Q190 set. "Cluster Architecture, Installation & Configuration" is worth 25% of the exam. The questions below fill that gap completely. Practice ALL of them — they are high-certainty exam tasks.

A. etcd Backup & Restore (near-certain exam task)

New Q1 — Critical
Backup etcd to a snapshot file
Take a snapshot of the etcd cluster running on the control plane. Save it to /opt/backup/etcd-snapshot.db. The etcd server uses TLS — cert, key, and CA files are at /etc/kubernetes/pki/etcd/.
⭐ Near-certain exam task. Must know: ETCDCTL_API=3 etcdctl snapshot save with --endpoints --cacert --cert --key flags.
New Q2 — Critical
Restore etcd from a snapshot
Restore etcd from snapshot at /opt/backup/etcd-snapshot.db to a new data directory /var/lib/etcd-restore. Update the etcd static pod manifest to use the new data directory and verify the cluster recovers.
⭐ Exam always pairs backup with restore. Must also know: how to update /etc/kubernetes/manifests/etcd.yaml --data-dir and hostPath volume.

B. RBAC (Role-Based Access Control)

New Q3 — Critical
Create Role and RoleBinding for a user
Create a Role pod-reader in namespace dev that allows get, list, watch on pods. Create a RoleBinding binding user jane to this role. Verify with kubectl auth can-i.
⭐ RBAC is 25% domain — always on exam. Must know both Role and RoleBinding YAML and the auth can-i --as verification.
New Q4 — Critical
Create ClusterRole and ClusterRoleBinding
Create a ClusterRole node-reader that allows get, list, watch on nodes. Bind it to ServiceAccount monitoring-sa in namespace monitoring using a ClusterRoleBinding.
⭐ Cluster-wide RBAC with ServiceAccount — common exam pattern. Know the difference: ClusterRoleBinding vs RoleBinding with ClusterRole.
New Q5 — Critical
Create a ServiceAccount and use it in a pod
Create ServiceAccount api-sa in namespace production. Create a pod that uses this ServiceAccount. Verify the token is mounted. Also create a pod with automountServiceAccountToken: false.
ServiceAccount usage in pods is frequently tested. Disabling automount is a security hardening task.
New Q6 — Important
Investigate and fix broken RBAC permissions
A pod running as ServiceAccount app-sa is getting "forbidden" errors trying to list ConfigMaps. Find the missing permission, create the appropriate Role and RoleBinding to fix it. Verify the pod can now access ConfigMaps.
RBAC troubleshooting is a common exam task — read errors, fix permissions, verify.

C. kubeadm — Cluster Setup & Upgrade

New Q7 — Critical
Bootstrap a cluster with kubeadm
Initialise a Kubernetes cluster using kubeadm init with pod network CIDR 10.244.0.0/16. Set up the kubeconfig. Install a CNI plugin (Flannel or Calico). Verify control plane pods are running.
⭐ kubeadm init is the entry point for the cluster architecture domain. Must know: --pod-network-cidr, copying kubeconfig, applying CNI manifest.
New Q8 — Critical
Join a worker node to the cluster
On a new worker node, join it to an existing cluster using a kubeadm join token. If the token has expired, generate a new one on the control plane. Verify the node appears and becomes Ready.
⭐ Worker node join is tested. Know: kubeadm token create --print-join-command and how to generate a discovery token hash.
New Q9 — Critical
Upgrade cluster from v1.29 to v1.30 using kubeadm
Upgrade the control plane: update apt repo, install kubeadm v1.30, run kubeadm upgrade apply v1.30.x, drain the control plane node, upgrade kubelet and kubectl, uncordon. Then upgrade a worker node: cordon, drain, upgrade kubelet, uncordon.
⭐ Cluster upgrade is a near-certain exam task. Know the exact sequence: kubeadm → drain → kubelet → uncordon.

D. Kubeconfig & Contexts

New Q10 — Critical
Manage kubeconfig: add cluster, user, context
Add a new cluster cluster-prod (API server https://192.168.1.10:6443, CA cert /opt/ca.crt) to kubeconfig. Add user admin-prod with client cert and key. Create context prod-context. Switch to it and verify with kubectl get nodes.
⭐ Every exam task begins with switching context/namespace. Must be fluent with kubectl config set-cluster/set-credentials/set-context/use-context.
New Q11 — Important
Merge two kubeconfig files and switch contexts
You have two kubeconfig files at ~/.kube/config and /opt/cluster2.yaml. Merge them using KUBECONFIG env variable. List all contexts, switch to the second cluster's context, and verify access.
Multi-cluster kubeconfig management is expected exam knowledge — KUBECONFIG=file1:file2 kubectl config view --flatten.

E. Node Maintenance

New Q12 — Critical
Safely drain a node for maintenance
Node worker-1 needs kernel maintenance. Cordon it so no new pods are scheduled. Drain it safely (handle DaemonSets and pods with local storage). Perform "maintenance" (echo done). Uncordon the node and verify pods reschedule.
⭐ Cordon/drain/uncordon is a guaranteed exam task. Know the flags: --ignore-daemonsets --delete-emptydir-data --force.
New Q13 — Important
Taint a node and verify scheduling behaviour
Add taint env=production:NoSchedule to node worker-2. Verify that a new pod without tolerations is not scheduled there. Create a pod with the correct toleration and verify it schedules on worker-2. Then remove the taint.
Applying and removing taints on nodes — your existing questions only cover tolerations, not tainting nodes.

F. Certificates

New Q14 — Important
Check and renew cluster certificates
Check the expiry of all kubeadm-managed certificates using kubeadm certs check-expiration. Renew all certificates with kubeadm certs renew all. Restart the control plane static pods and verify the API server is accessible.
Certificate renewal is tested — especially in troubleshooting scenarios where the API is unreachable due to expired certs.
New Q15 — Important
Create a CertificateSigningRequest for a new user
Generate a private key and CSR for user john. Create a Kubernetes CertificateSigningRequest object, approve it, and extract the signed certificate. Configure a kubeconfig entry for john using the signed cert.
CSR workflow (key → CSR → CertificateSigningRequest → approve → cert) is an exam task in the RBAC/auth domain.

G. kubectl Output & JSONPath

New Q16 — Critical
Use JSONPath to extract specific fields
Using a single kubectl command: (a) get the IP of all nodes, (b) list all pod names and their node assignments, (c) get the image used by all containers in all pods in namespace default. Use -o jsonpath.
⭐ JSONPath output is tested in almost every CKA exam. Know: '{range .items[*]}{.metadata.name}{"\t"}{.status.podIP}{"\n"}{end}' pattern.
New Q17 — Important
Sort and filter kubectl output
List all pods across all namespaces sorted by creation time. List all PVs sorted by capacity. Find all pods that are NOT in Running state. Output results to files using --sort-by and -o custom-columns.
Sorting and filtering output is a common exam requirement — often paired with "save the output to file /opt/answer.txt".

H. Static Pods

New Q18 — Important
Create a static pod on a specific node
Create a static pod named static-web running nginx:1.21 on node worker-1. SSH to the node, place the pod manifest in /etc/kubernetes/manifests/. Verify the pod appears in kubectl get pods. Then delete it by removing the manifest file.
Static pods are a cluster architecture concept — tested in setup and troubleshooting scenarios.

I. Pod Security Standards

New Q19 — Important
Apply Pod Security Standards to a namespace
Label namespace secure-ns to enforce the baseline pod security standard and warn on restricted. Try to create a privileged pod — verify it is rejected. Create a non-privileged pod — verify it is allowed.
PSS replaced PodSecurityPolicy (PSP) in Kubernetes 1.25+. Current exam uses PSS. Know the three levels: privileged, baseline, restricted.

J. Imperative Commands Mastery (Exam Speed)

New Q20 — Critical
Create resources imperatively and save as YAML
Using only imperative commands (no writing YAML by hand), create: (a) a deployment myapp with image nginx and 3 replicas, (b) expose it as a ClusterIP service on port 80, (c) create a ConfigMap from literal values, (d) create a Secret with username/password. Save each as YAML with --dry-run=client -o yaml.
⭐ Speed is critical in a 2-hour exam. Being able to generate YAML imperatively is 10x faster than writing it from scratch.
New Q21 — Critical
Rollout management: update, rollback, history
Update deployment webapp image to nginx:1.22 with a change-cause annotation. Check rollout status. Simulate a bad update (nginx:broken) and watch it fail. Roll back to the previous version. View rollout history and roll back to a specific revision.
⭐ Rollout and rollback tasks appear on every CKA exam — kubectl rollout status/history/undo --to-revision.
New Q22 — Important
Scale deployments and configure HPA imperatively
Scale deployment api-server to 5 replicas imperatively. Create an HPA for it: min 2, max 10, CPU target 70%. Verify HPA is created. Use kubectl autoscale command (not YAML).
Imperative scaling and autoscaling — know kubectl scale and kubectl autoscale.

K. ConfigMaps & Secrets (if not covered in earlier questions)

New Q23 — Important
ConfigMap and Secret usage patterns
Create a ConfigMap from a file, from literals, and from a directory. Create a Secret (generic, TLS, docker-registry). Use them in a pod as: (a) environment variables, (b) volume mounts. Verify the values are accessible inside the container.
ConfigMap/Secret mounting is in every exam — both env var and volume patterns must be fluent.

L. Additional Troubleshooting (Cluster Architecture)

New Q24 — Critical
Fix a broken control plane component (static pod)
The kube-scheduler is not running. Investigate: check /etc/kubernetes/manifests/kube-scheduler.yaml for misconfigurations (wrong flag, bad image, incorrect volume path). Fix the manifest and verify the scheduler restarts and pods begin scheduling.
⭐ Breaking and fixing static pod manifests for control plane components is a very common exam task. The exam deliberately introduces errors in manifest files.
New Q25 — Critical
Fix a broken kubelet configuration on a worker node
Worker node worker-1 is NotReady. SSH to the node. The kubelet service has a wrong --config path or a typo in /var/lib/kubelet/config.yaml. Fix the configuration, restart kubelet with systemctl restart kubelet, verify the node becomes Ready.
⭐ Broken kubelet is the #1 most common troubleshooting task on the CKA exam. Practise this until it's muscle memory.
New Q26 — Important
Identify and fix a pod with wrong node selector / node name
A pod is stuck Pending because it has nodeName: worker-99 (doesn't exist) or a nodeSelector label that no node has. Identify the issue, fix the pod spec, and verify it schedules correctly.
Wrong nodeName/nodeSelector is a deliberate exam trap — the fix requires editing the pod or deleting/recreating it.
New Q27 — Important
Use crictl to debug containers when kubectl is unavailable
On a node where kubectl is not available (or API server is down), use crictl ps to list containers, crictl logs <id> to get container logs, crictl inspect <id> to inspect container details. Identify a misconfigured container.
When the API server is down (as in etcd-restore scenarios), crictl is the only tool available. Know: crictl ps -a, crictl logs, crictl rmi.
New Q28 — Important
Investigate events and resource status across the cluster
Use kubectl get events --sort-by=.lastTimestamp -A to find recent cluster events. Identify pods with Warning events. Use kubectl describe node to read node conditions and allocated resources. Save events to a file.
Events are the first place to look when something is wrong — exam tasks often require reading events to identify the root cause.
New Q29 — Good to Know
Network troubleshooting with a debug pod
Deploy a debug pod (nicolaka/netshoot or busybox) in a namespace. Use it to: run nslookup for service DNS, curl a service endpoint, ping another pod IP, check /etc/resolv.conf. Use kubectl exec and kubectl run --rm -it patterns.
In-cluster debugging with a temporary pod is an essential exam technique — many network tasks require running commands from inside a pod.
New Q30 — Good to Know
Verify cluster component health comprehensively
Systematically check: kubectl get componentstatuses, all pods in kube-system, node conditions, API server health endpoint /healthz, etcd member list, kubelet service status on each node. Document any anomalies.
Cluster health check is both an exam task type and a real-world skill. Know all the check commands so you can quickly triage any scenario.
